Рефлексия
1. Какие функции войдут в публичный интерфейс множества, а какие останутся деталями реализации? Правда ли они могут иметь другое название?

В публичный интерфейс должны войти только те функции, которые описывают поведение множества как абстрактного типа данных, независимо от способа его внутреннего хранения. К таким операциям относятся: bstInsert, bstContains, bstRemove, bstSize, bstFree, bstMerge, bstIteratorInit, bstIteratorHasNext, bstIteratorNext, bstIteratorFree.

Все эти функции имеют смысл для любого множествах, поэтому названия могут быть изменены на более общие, не привязанные к конкретной структуре данных: bstInsert -> setInsert, bstContains -> setContains, bstRemove -> setRemove, bstSize -> setSize, bstFree -> setFree, bstMerge -> setMerge, bstIteratorInit -> setIteratorInit, setIteratorHasNext -> setIteratorHasNext, bstIteratorNext -> setIteratorNext, bstIteratorFree -> setIteratorFree.

Остальные функции являются внутренними и не должны быть доступны пользователю множества. Они либо вспомогательные, либо отражают специфику бинарного дерева поиска: bstNodeFree, bstInorder, bstPreorder, bstPostorder, bstInorderInner, bstPreorderInner, bstPostorderInner, bstHeight, bstNodeHeight, bstMin, bstMax, bstKthMin, bstKthMinInner, bstIsValid, bstIsValidInner, bstNextNodeInorder.

Эти функции не должны присутствовать в публичном заголовочном файле, если мы хотим сохранить возможность замены реализации.

2. Какие из написанных вами функций не имеют смысла в интерфейсе множества?
Исходя из анализа выше, следующие функции не имеют смысла в интерфейсе абстрактного множества: они привязаны к упорядоченности, или являются деталями реализации:

bstInorder, bstPreorder, bstPostorder, bstMin, bstMax, bstHeight, bstKthMin, bstIsValid, все функции с суффиксом Inner (они вспомогательные и не предназначены для вызова извне), bstNodeFree.

3. Что нужно изменить, чтобы можно было безболезненно заменить BST на другую реализацию (например, хеш-таблицу), не меняя код, который использует множество?

1) В заголовочном файле объявить typedef struct Set Set; без определения полей. Все функции должны принимать и возвращать указатели на Set. Внутренняя структура (BST) определяется только в файле реализации.

2)
Переименовать публичные функции, убрав приставку bst и заменив на set.

3) Убрать из заголовочного файла все функции, перечисленные в вопросе 2. При этом реализация итератора для BST будет использовать inorder обход (если мы хотим сохранить упорядоченность) или любой другой. Для общего множества порядок обычно не гарантируется, поэтому лучше сделать итератор, обходящий элементы в произвольном порядке.

4) Весь код, специфичный дя BST (структура узла, вспомогательные функции), перенести в set.c и сделать, чтобы они были видны только в пределах этого файла.
